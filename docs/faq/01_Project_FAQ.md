# 1. Project Overview

## Q1. What is RideNow?

RideNow is a fictional ride-hailing company created to demonstrate a complete enterprise Data Engineering platform using Snowflake, SQL, Python, Power BI, and GitHub.

The project simulates a real-world production environment by implementing enterprise architecture, modern ETL/ELT pipelines, dimensional modeling, and comprehensive documentation.

---

## Q2. Why was this project created?

The primary objective is to demonstrate practical knowledge of enterprise Data Engineering rather than isolated coding exercises.

The project showcases:

- Enterprise Architecture
- Snowflake Data Warehouse
- Data Modeling
- SQL Development
- Python Data Generation
- ETL/ELT Pipelines
- Power BI Reporting
- GitHub Documentation
- Enterprise Coding Standards

---

## Q3. Is RideNow a real company?

No.

RideNow is a fictional organization inspired by ride-hailing businesses such as Uber and Ola. It provides a realistic business domain for demonstrating enterprise data engineering concepts.

---

## Q4. Who is the target audience for this project?

This repository is intended for:

- Data Engineers
- Snowflake Developers
- Data Architects
- Business Intelligence Developers
- Hiring Managers
- Technical Interviewers
- Students learning Data Engineering

# 2. Business Scenario

## Q5. What business problem does RideNow solve?

RideNow centralizes operational data generated by customer applications, driver applications, payment gateways, GPS systems, and promotional services into a unified analytics platform.

The platform enables:

- Business reporting
- Performance monitoring
- Customer analytics
- Driver analytics
- Revenue analysis
- Operational dashboards

---

## Q6. Why choose a ride-hailing domain?

Ride-hailing platforms generate diverse datasets including:

- Customer information
- Driver information
- Vehicle details
- Trips
- Payments
- Ratings
- Promotions
- GPS data

This makes the domain ideal for demonstrating enterprise data engineering patterns.

# 3. Architecture

## Q7. Which architecture does the project follow?

The project follows the Medallion Architecture:

- Bronze
- Silver
- Gold

Each layer has a clearly defined responsibility.

---

## Q8. Why use Medallion Architecture?

Benefits include:

- Improved scalability
- Better data quality
- Clear separation of concerns
- Simplified maintenance
- Reusable transformations
- Faster reporting

# 5. Data Modeling

## Q11. Which modeling approach is used?

The project uses:

- Enterprise Data Modeling
- Star Schema
- Fact Tables
- Dimension Tables
- Surrogate Keys
- Business Keys
- Slowly Changing Dimensions (SCD)

## Q12. Why use Surrogate Keys?

Surrogate keys improve:

- Join performance
- Historical tracking
- ETL simplicity
- Warehouse consistency
